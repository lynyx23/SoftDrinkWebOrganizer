Abstract

This document outlines the system architecture, requirements, and structural specifications for SOr (Soft Drink Web Organizer), commercially styled as The Soda Aisle. SOr is a modern, responsive Web application designed to manage personal and group preferences for non-alcoholic beverages, including teas, dairy, juices, and syrups. The software is built fully from scratch using vanilla PHP 8.x on the backend and native JavaScript/CSS/HTML on the frontend. It deliberately circumvents the use of third-party frameworks to adhere strictly to strict academic engineering constraints, highlighting native Web APIs, secure database models, and service-oriented communication.

1. Introduction

1.1 Purpose

The purpose of this document is to detail the technical framework, system behavior, and data modeling of the SOr platform. It acts as both a system requirements specification guide and an architectural blueprint for developers, system administrators, and evaluators.

1.2 Document Conventions

This technical report follows the IEEE System Requirements Specification (SRS) structure template and is encoded using the Scholarly HTML standard for structured semantic discovery.

1.3 Intended Audience and Reading Suggestions

This documentation is intended for Web development professionals, academic evaluators, and system maintainers. It is recommended to read Section 3 for specific product behaviors and Section 5 for the underlying database schema and security implementations.

1.4 Product Scope

SOr serves as an advanced centralized repository and preference log for non-alcoholic drinks. It supports multi-session user state control, an automated "Safe Mode" catalog filter based on custom dietary restriction rules, group-based social workspaces, shopping cart list builders, analytical data exports via direct stream generation (CSV and SVG formats), and global RSS popular product rankings.

2. Overall Description

2.1 Product Perspective

SOr operates as an independent client-server system communicating entirely over asynchronous RESTful application services. The backend exposes secure JSON endpoints through a unified routing unifier (Front Controller pattern), while the frontend leverages native asynchronous Fetch API calls (AJAX), ensuring an optimal single-page feel without structural page reloads.

2.2 Product Functions

The primary macro-functions implemented within the system are:

  • Secure Session Controls: Coded multi-device authentication tokens stored in an isolation layer.
  • Allergen & Diet-Safe Catalog: Excludes foods or drinks matching an active consumer's allergy tags while strictly filtering by structural diets (e.g., Vegan, Vegetarian).
  • Group Coordination Systems: Allows social circles to merge preference models, view aggregated stats, and compile cohesive requests.
  • Shopping List Workspaces: Real-time co-authoring tools to manage shopping carts with bought/pending item toggles.
  • Analytic Engine: Formats raw metrics directly into Open Data standards (tabular CSV arrays and dynamically drawn SVG charts).
  • Open Food Facts Proxy Handler: Real-time barcode ingestion mapping ingredients, macros, and certifications straight from public databases.

2.3 User Classes and Characteristics

User Class Privileges & Context
Guest (Anonymous) Can browse public listings, review store statistics, and subscribe to the RSS syndication stream.
User (Authenticated) Maintains a health/diet profile, saves ratings, manages personal/group carts, and submits product suggestions.
Admin (Moderator) Full access to underlying records, manages user lifecycles, and recenses or updates pending catalog submissions.

2.4 Design and Implementation Constraints

The engine is restricted to vanilla PHP 8.x, utilizing an explicit object-oriented architecture, the Singleton pattern for persistence connectivity, and an inline MVC design pattern. Data storage relies entirely on SQLite 3 for localized, zero-overhead execution.

3. System Features & Technical Deep-Dive

3.1 Authentication & Multi-Session Security

User lifecycle logic is managed by the AuthController. Account passwords undergo secure one-way hashing using the Bcrypt algorithm via native password_hash() APIs. Upon successful token exchange at POST /api/users/login, existing tokens are flushed, and a clean cryptographic token (64 hexadecimal characters via bin2hex(random_bytes(32))) is provisioned to client-side localStorage, passed via the Authorization: Bearer <token> header on subsequent protected API contexts.

3.2 Smart Catalog Filtering (Safe Mode Interceptor)

The GET /api/beverages endpoint handles query-string filters alongside a boolean safe_mode=1 switch. When invoked by an authenticated consumer, the system queries user restrictions to modify underlying query pipelines: 

-- Exclude allergen cross-matches & enforce diet matching rules
SELECT * FROM beverages WHERE 1=1
AND id NOT IN (
    SELECT br.beverage_id
    FROM beverage_restrictions br
    JOIN restrictions r ON br.restriction_id = r.id
    JOIN user_restrictions ur ON ur.restriction_id = r.id
    WHERE ur.user_id = :user_id AND r.type = 'allergen'
)
AND NOT EXISTS (
    SELECT 1 FROM user_restrictions ur2
    JOIN restrictions r2 ON ur2.restriction_id = r2.id
    WHERE ur2.user_id = :user_id AND r2.type = 'diet'
    AND ur2.restriction_id NOT IN (
        SELECT br3.restriction_id
        FROM beverage_restrictions br3
        WHERE br3.beverage_id = beverages.id
    )
);

3.3 Analytical Data Pipelines (CSV & SVG Real-time Rendering)

Handled inside StatsController::export, the application generates live binary/text stream attachments written directly to the PHP active output buffer, preventing disk overhead or permission vulnerabilities:

  • CSV Pipeline: Emits structurally aligned matrices containing beverage identities, categories, aggregated average rating figures, and transaction volume votes.
  • SVG Chart Generator: Compiles an inline XML graphics layout rendering horizontal trend data for the top 5 highest-rated stock elements. The code utilizes native math translations to construct desaturated retro color fields matching the theme.

3.4 RSS Syndication Feed

The RssController::feed maps data straight into an application/rss+xml stream. It loops over the 10 highest-ranked catalog beverages, converting product descriptions, categories, and cumulative satisfaction metrics into an open RSS 2.0 valid data flux.

4. External Interface & Open Food Facts Integration

4.1 User Interface & Client DOM Lifecycles

The visual layer honors an explicit mid-century mercantile motif defined across modular layout stylesheets (`base.css`, `style.css`). Layout grids rely heavily on flexible, responsive blocks. Client elements avoid inline script listeners, deploying semantic event delegation models that process network states asynchronously.

4.2 Server-Side cURL Open Food Facts Proxy Interceptor

To prevent client-side Cross-Origin Resource Sharing (CORS) rejections and guarantee strict compliance with OFF data policies, product fetches operate via a backend routing tunnel inside `BeverageController::lookupOff`. It sets compliant `User-Agent` markers and queries the public open database: 

// Server-Side proxy lookup snippet
$offUrl = "https://world.openfoodfacts.org/api/v0/product/{$barcode}.json";
$ch = curl_init();
curl_setopt($ch, CURLOPT_URL, $offUrl);
curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
curl_setopt($ch, CURLOPT_HTTPHEADER, ["User-Agent: SoftDrinkWebOrganizer/1.0 (tudorbc23@gmail.com)"]);
curl_setopt($ch, CURLOPT_SSL_VERIFYPEER, false);
$result = curl_exec($ch);

The resulting macro array parses structural nutrient components and cross-references labels, automatically checking relevant database dietary restrictions before serving the unified model back to the view layer.

5. Database Architecture & Defensive Security Models

5.1 Structural Schema Overview

The system maps relations across relational schema architectures within a single, secure file context:

-- Simplified Relational Database Setup
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    email TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    role TEXT NOT NULL DEFAULT 'user' CHECK (role IN ('user', 'admin')),
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE beverages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    barcode TEXT UNIQUE,
    category TEXT,
    price REAL NOT NULL CHECK (price >= 0),
    volume_ml INTEGER,
    nutritional_info TEXT, -- Formatted JSON text string
    nutriscore TEXT CHECK (LOWER(nutriscore) IN ('a', 'b', 'c', 'd', 'e'))
);

CREATE TABLE preferences (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    group_id INTEGER,
    beverage_id INTEGER NOT NULL,
    rating INTEGER CHECK (rating BETWEEN 1 AND 5),
    notes TEXT,
    FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE,
    FOREIGN KEY (beverage_id) REFERENCES beverages (id) ON DELETE CASCADE
);

5.2 Structural Security & Cyber Defense Mitigation Layers

Defensive design constraints are handled via strict operational layers:

  • SQL Injection Shielding: Raw statements are strictly banned. Database actions bind criteria values exclusively inside PDO PreparedStatement execution tokens.
  • Cross-Site Scripting (XSS) Sanitizer: The Security::sanitize() unifier deeply intercept arrays, converting incoming properties through htmlspecialchars((string) $data, ENT_QUOTES, 'UTF-8') ensuring pure execution contexts inside client-side templates.

6. Conclusions

SOr (The Soda Aisle) illustrates that constructing standard-compliant service architectures is highly viable and performant using pure, vanilla software engineering practices. By avoiding heavy modular abstractions, the application delivers optimal resource consumption, airtight execution contexts, and low maintenance friction.
